# /learn-pc-write — ゼロから書く（01-03）

テーマを選び、タスク記述をゼロから 3 件作成する演習です。

**所要時間**: 約 60 分
**前提**: `/learn-pc-rewrite` 完了推奨（必修ではない）
**スキル対応**: Prompt Craft（指示の構造化）

---

## Step 1: テーマ選択

```
「ここまでは既存のタスクを「改善」してきました。
この演習では、ゼロから自分でタスクを書きます。

以下のテーマから 1 つ選ぶか、自分の業務から題材を持ってきてください:

A. 社内の備品管理アプリ
B. チーム用の日報ボット
C. ファイルの自動バックアップスクリプト
D. 自分の業務から（自由テーマ）

テーマ D を強く推奨します。
自分の業務に近いほど、学びが実務に直結します。」
```

---

## Step 2: 成果物の定義

選んだテーマで成果物（Deliverable）を定義する。

```
「まず、このテーマの成果物を定義しましょう。
成果物はタスクをグループ化したもので、「何を作るか」の全体像です。

タイトルと説明を考えてください:
- タイトル: 短く明確に（例: 「備品管理 Web アプリ MVP」）
- 説明: このプロダクト/機能の目的と主な利用者」
```

受講者の回答を受けて `upsert_deliverable(projectId, title, description)` で登録。

---

## Step 3: タスク 3 件を作成

```
「この成果物を実現するために、3 つのタスクに分解してください。

各タスクに以下の 4 要素を含めてください:
- ## 背景 — なぜこのタスクが必要か
- ## 対応内容 — 何をするか（箇条書きで具体的に）
- ## 完了条件 — チェックリスト形式で検証可能な基準
- ## 制約 — やらないこと・スコープ外

まず 1 件目から始めましょう。タイトルと内容を教えてください。」
```

各タスクについて:

1. 受講者がタイトルと description を考える
2. 内容を確認し、不足があれば問いかけで気づかせる（直接指摘しない）:
   - 「このタスクの完了をどうやって確認しますか？」
   - 「このタスクでやらないことは何ですか？」
   - 「具体的にどのファイル・どの機能を触りますか？」
3. `upsert_task(projectId, title, deliverableId, description)` で登録

**ペース**: 1 件目は丁寧にフィードバック、2〜3 件目は受講者のペースで。

---

## Step 4: 自己レビュー

```
「3 件のタスクが完成しました！
品質をセルフチェックしてみましょう。」
```

作成した 3 件に対して `/review-task` の観点でスコアリング:

| チェック観点         | タスク1 | タスク2 | タスク3 |
| -------------------- | ------- | ------- | ------- |
| 背景（なぜやるか）   | ○/△/×   | ○/△/×   | ○/△/×   |
| 対応内容の具体性     | ○/△/×   | ○/△/×   | ○/△/×   |
| 完了条件（検証可能） | ○/△/×   | ○/△/×   | ○/△/×   |
| 制約（やらないこと） | ○/△/×   | ○/△/×   | ○/△/×   |

- 全項目 ○ → 「素晴らしい！Prompt Craft の基本が身についています」
- △ がある → 該当箇所の改善を提案（`upsert_task` で更新）
- × がある → 具体的にどう改善するか一緒に考える

---

## Step 5: 振り返り

```
「ゼロから書くのは、改善するより難しかったですか？

多くの受講者が感じるポイント:
- 改善は「何が足りないか」が見えやすい
- ゼロからだと「何を書くべきか」の判断が必要
- 制約（やらないこと）を定義するのが意外と難しい

これは AI への指示でも同じです。
「何をしてほしいか」だけでなく「何をしてほしくないか」を
明示することで、AI の出力精度は大幅に上がります。

次の /learn-pc-review では、他の人が書いたタスクに
フィードバックを出す練習をします。
人に指摘することで、自分の基準もさらに明確になります。」
```

---

## 注意事項

- テーマ D（自由テーマ）を強く推奨する。実務に近いほど学びが深い
- 受講者が完璧を目指しすぎないよう、「まず書いてみて、あとで改善」を促す
- 3 件のタスクは互いに独立していなくてもよい（連続する作業の分割でも OK）
- 成果物とタスクの紐づけ（deliverableId）を忘れない
