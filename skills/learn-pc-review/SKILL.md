# /learn-pc-review — 他者レビュー（01-04）

他の学習者（またはサンプル）のタスク記述を読み、フィードバックを付ける演習です。

**所要時間**: 約 30 分
**前提**: `/learn-pc-write` 完了推奨（必修ではない）
**スキル対応**: Prompt Craft（指示の構造化）

---

## Step 1: 導入

```
「Prompt Craft は自分で書くだけでなく、
他者の記述をレビューする力も含みます。

レビューは AI への指示だけでなく、
チーム内のコミュニケーション品質を高めるスキルです。

この演習では、タスク記述を読んでフィードバックを
ProjSight の Question として記録します。」
```

---

## Step 2: レビュー対象の取得

### パターン A: 複数の学習者がいる場合

1. 他の学習者のプロジェクトから `list_tasks` でタスクを取得
2. `/learn-pc-write` で作成された 2 件を選ぶ

### パターン B: 単独受講の場合

1. `.claude/skills/learn-data/medium-tasks.json` を Read ツールで読み込む
2. JSON の各オブジェクトの `title` と `description` を受講者に提示する
3. `issues` と `improvementHints` はフィードバック時の参考情報（受講者には見せない）
4. 必要に応じて `upsert_task` で学習プロジェクトに投入してからレビューする

中品質タスク記述 2 件をレビュー対象とする。これらは構造（背景・対応内容・完了条件）はあるが品質が不十分な記述で、受講者が具体的な改善点を見つけるのに適している。

---

## Step 3: フィードバック作成

各タスクについて以下の手順で進める:

1. **品質チェック**: 以下の 4 観点で評価

| チェック観点     | 確認内容                               | 評価  |
| ---------------- | -------------------------------------- | ----- |
| 背景             | 「なぜやるか」が具体的に書かれているか | ○/△/× |
| 対応内容の具体性 | 変更箇所・操作が特定できるか           | ○/△/× |
| 完了条件         | Yes/No で検証できるか                  | ○/△/× |
| 制約             | やらないことが明記されているか         | ○/△/× |

2. **改善提案の作成**: 具体的な修正案を含むフィードバックを書く

```
「このタスク記述のフィードバックを書いてみてください。

フィードバックの構成:
1. 良い点（まず褒める）
2. 改善すべき点（4 観点のどれに該当するか明記）
3. 具体的な修正案（こう書き換えるとよい）」
```

3. **ProjSight への記録**: `upsert_question(projectId, title, body)` でフィードバックを記録
   - title: `タスク「{タスク名}」へのレビューフィードバック`
   - body: 受講者が書いたフィードバック内容

---

## Step 4: 振り返り

```
「2 件のレビューを通じて、いくつかの気づきがあったと思います。

レビューの効果:
1. 人の記述の問題点を指摘することで、自分の基準が明確になる
2. 「具体的な修正案」を書く訓練は、自分の記述力を高める
3. 良い点を見つける力は、AI の出力を評価する力にもつながる

これは AI への指示だけでなく、人間へのフィードバックにも通じます:
- コードレビューで「ここがダメ」ではなく「こう書くとよい」
- 仕様確認で「曖昧です」ではなく「○○の場合はどうなりますか？」

次の /learn-pc-reporting では、
Prompt Craft を対人コミュニケーション（報告・連絡・相談）に応用します。」
```

---

## 注意事項

- フィードバックは「批判」ではなく「改善提案」であることを強調する
- 良い点を先に見つけてから改善点を指摘する習慣をつける
- 複数の学習者がいる場合、匿名レビュー（誰のタスクかを伏せる）も検討
- サンプルの「隠れた問題点」は受講者に直接見せない。自分で発見させる
