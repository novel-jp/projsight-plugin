# /learn-se-decompose — 成果物→タスク分解（04-01、必修）

成果物（Deliverable）を定義し、独立して検証可能なタスクに分解する演習です。

**所要時間**: 約 60 分
**前提**: `/learn-ie-analyze`（03-01）完了済み
**スキル対応**: Specification Engineering（自律実行できる仕様）

---

## Step 1: 導入

```
「Specification Engineering — AI が数日から数週間、迷わず実行できる『一貫した仕様』を書く最上位スキルです。

Prompt Craft（何を伝えるか）、Context Eng.（どんな情報環境を作るか）、Intent Eng.（なぜを記録するか）
の 3 スキルを統合し、『自律実行できる仕様』を構築します。

このセクションでは以下を学びます:
- 成果物をタスクに分解する技術
- 各タスクに完了条件を設定する方法
- AI が迷わず実行できるタスク記述の書き方」
```

---

## Step 2: 成果物の定義

受講者に 1 つの成果物（Deliverable）を定義してもらう。テーマ例:

| テーマ                  | 概要                                       |
| ----------------------- | ------------------------------------------ |
| **A**: ユーザー認証機能 | ログイン・サインアップ・パスワードリセット |
| **B**: ブログ機能       | 記事の作成・編集・削除・一覧表示           |
| **C**: ダッシュボード   | KPI カード・グラフ・フィルタ機能           |
| **D**: 自由テーマ       | 受講者のプロジェクトから                   |

```
「まず成果物を 1 つ定義しましょう。
成果物とは『プロジェクトの最終的な出力物』です。
テーマを選んでください。自分のプロジェクトの成果物でも構いません。」
```

受講者が選んだら `upsert_deliverable` で ProjSight に登録する。

---

## Step 3: タスク分解

```
「この成果物を 4〜6 個の独立したタスクに分解してください。

分解の基準:
- 1人が1セッションで完了できるサイズ
- 独立して検証可能 — 他のタスクに依存せず動作確認できる
- 完了条件が明確 — 「できた/できてない」が客観的に判定できる」
```

各タスクに以下を含めるよう誘導:

| フィールド   | 説明                   | 例                                                         |
| ------------ | ---------------------- | ---------------------------------------------------------- |
| **背景**     | なぜこのタスクが必要か | 「ユーザーがパスワードを忘れた場合に...」                  |
| **対応内容** | 具体的に何をするか     | 「メール送信API + リセット画面を実装」                     |
| **完了条件** | 検証可能な基準         | 「リセットメールが送信され、新パスワードでログインできる」 |
| **制約**     | 守るべきルール         | 「トークンの有効期限は24時間」                             |

受講者が各タスクを定義したら、1つずつ `upsert_task(deliverableId)` で登録する。

---

## Step 4: /review-task で品質チェック

登録したタスクを `/review-task` スキルでレビューする。

- 各タスクのスコアを確認
- **8 点未満**のタスクは改善を提案
- 特に以下を重点チェック:
  - 完了条件が検証可能か
  - タスクが独立しているか（他タスクへの暗黙の依存がないか）
  - 対応内容が具体的か（「〜を実装する」だけでなく詳細があるか）

改善が必要な場合は受講者に修正してもらい、`upsert_task` で更新する。

---

## Step 5: 振り返り

```
「タスク分解は Specification Engineering の基礎です。

ポイント:
- 『独立して検証可能なサイズ』が鍵 — 大きすぎるタスクは AI も人も迷う
- 完了条件は『動詞 + 目的語 + 検証方法』で書く
- 制約の明記が AI の暴走を防ぐ — 制約なしは『何でもあり』と同じ
- 良いタスク分解は、そのまま AI への実装指示になる

次の /learn-se-workflow では、ここで作ったタスクを実際に start_work → complete_work で遂行します。」
```

受講者のタスクを `upsert_task(progressPct: 100)` で完了にする。
