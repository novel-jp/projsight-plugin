# /learn-pc-rewrite — タスク記述の改善（01-02）

「悪いタスク記述」を読み、品質基準に基づいて改善版を書く演習です。

**所要時間**: 約 45 分
**前提**: `/learn-pc-compare` 完了済み
**スキル対応**: Prompt Craft（指示の構造化）

---

## Step 1: 導入

```
「ProjSight に登録された 5 件のタスク記述を読んで、改善版を書く演習です。

各タスクの description を読み、以下の 4 観点で改善してください:
✅ 背景（なぜやるか）
✅ 対応内容（具体的に何をするか）
✅ 完了条件（検証可能な基準）
✅ 制約（やらないこと）

前回の比較演習で体感した「構造化された指示」の力を、
タスク記述に応用する練習です。」
```

---

## Step 2: 悪いタスク記述 5 件を提示

### データの読み込み

1. `.claude/skills/learn-data/bad-tasks.json` を Read ツールで読み込む
2. JSON の各オブジェクトの `title` と `description` を使用する
3. `issues` と `learningPoint` はフィードバック時の参考情報（受講者には見せない）
4. `exampleFix` は受講者が改善案を書いた後の比較用（Step 3 で必要に応じて提示）

### サンプルタスクの投入

学習プロジェクトのタスクを `list_tasks(projectId)` で取得し、改善対象のタスクを確認する。

**テンプレートプロジェクトにタスクが配置されていない場合**、`bad-tasks.json` の 5 件をサンプルとして `upsert_task` で学習プロジェクトに登録してから改善演習を行う:

```
「改善対象のサンプルタスクをプロジェクトに登録します。
登録後、1 件ずつ description を改善していきましょう。」
```

既に同名タスクが存在する場合はスキップする。

---

## Step 3: 1 件ずつ改善

各タスクについて以下の手順で進める:

1. **問いかけ**: 「このタスク記述の問題点は何ですか？」
2. **受講者の回答を待つ**
3. ヒントが必要なら 4 観点のチェックリストを再提示:
   - 背景: 「なぜやるか」が書かれているか？
   - 対応内容: 具体的な変更箇所・操作が書かれているか？
   - 完了条件: Yes/No で判定できる検証項目があるか？
   - 制約: やらないこと・スコープ外が明記されているか？
4. **受講者に改善版を考えてもらう**
5. 改善版を確認し、フィードバックを提供
6. ProjSight にタスクが存在する場合は `upsert_task(taskId, description)` で改善版を上書き

### 模範解答の参照（受講者が困っている場合のみ提示）

`bad-tasks.json` の該当タスクの `exampleFix.description` を参照し、模範解答として提示する。
模範解答は「唯一の正解」ではないことを伝え、受講者自身の改善版と比較させる。

**ペース配分**: 5 件全部をじっくりやると時間がかかるため、最初の 2 件は丁寧に、残り 3 件はテンポよく進める。

---

## Step 4: /review-task で自己評価

改善した 5 件（または受講者が改善版を書いたもの）について品質チェックを実施する。

```
「改善したタスク記述の品質を `/review-task` の観点でチェックしてみましょう。」
```

各タスクの description を以下の観点で評価:

| チェック観点     | 確認内容                                                   |
| ---------------- | ---------------------------------------------------------- |
| 背景             | `## 背景` セクションがあり、「なぜやるか」が書かれているか |
| 対応内容の具体性 | 具体的な変更箇所・操作が箇条書きで書かれているか           |
| 完了条件         | チェックリスト形式で、検証可能（Yes/No で判定できる）か    |
| 制約             | やらないこと・スコープ外が明記されているか                 |

改善前後でどこが良くなったかを具体的にフィードバックする。

---

## Step 5: 振り返り

```
「5 件の改善を通じて、良いタスク記述のパターンが見えてきましたか？

共通するポイント:
1. 「おかしい」「遅い」「足りない」→ 具体的に何が、どう問題なのかを書く
2. 「直す」「改善する」「追加する」→ 具体的にどのファイル、どの関数、何をするかを書く
3. 完了条件は「〜できること」ではなく「〜した結果、〜が確認できること」
4. やらないことを明記することで、スコープの膨張を防ぐ

次の /learn-pc-write では、ゼロから自分でタスクを書きます。
改善するより難しいですが、ここで学んだパターンを活かしてください。」
```

---

## 注意事項

- 受講者自身に問題点を見つけてもらうことが重要。すぐに答えを教えない
- 模範解答は「唯一の正解」ではないことを伝える
- 受講者の業務経験に応じて、より実践的な例を追加してもよい
- テンプレートプロジェクトにタスクが配置されている場合はそちらを優先する
