# /learn-pc-rewrite — タスク記述の改善（01-02）

「悪いタスク記述」を読み、品質基準に基づいて改善版を書く演習です。

**所要時間**: 約 45 分
**前提**: `/learn-pc-compare` 完了済み
**スキル対応**: Prompt Craft（指示の構造化）

---

## Step 1: 導入

```
「ProjSight に登録された 4 件のタスク記述を読んで、改善版を書く演習です。

各タスクの description を読み、以下の 4 観点で改善してください:
✅ 背景（なぜやるか）
✅ 対応内容（具体的に何をするか）
✅ 完了条件（検証可能な基準）
✅ 制約（やらないこと）

前回の比較演習で体感した「構造化された指示」の力を、
タスク記述に応用する練習です。」
```

---

## Step 2: 悪いタスク記述 4 件を提示

### データの読み込み

1. `.claude/skills/learn-data/bad-tasks.json` を Read ツールで読み込む
2. JSON の各オブジェクトの `title` と `description` を使用する
3. `issues` と `learningPoint` はフィードバック時の参考情報（受講者には見せない）
4. `exampleFix` は受講者が改善案を書いた後の比較用（Step 3 で使用）
5. **bad-5 は使用しない**（bad-2 と学習ポイントが重複するため除外）

### 演習順序

学習効果を考慮し、以下の順番で進める:

1. **bad-1** — 最も典型的な「曖昧さ」のパターン
2. **bad-3** — 権限・ロール設計を含み、構造化の深掘りに適している
3. **bad-2** — 主観的な表現を定量化するパターン
4. **bad-4** — テスト追加タスクの構造化

### サンプルタスクの投入

`/learn-setup` で作成した学習プロジェクトに演習用タスクを登録する。

学習プロジェクトの projectId は `list_projects` で取得し、`list_tasks(projectId)` で既存タスクを確認する。

**学習プロジェクトにタスクが配置されていない場合**、`bad-tasks.json` の 4 件（bad-1〜bad-4）を `upsert_task` で学習プロジェクトに登録してから改善演習を行う:

```
「改善対象のサンプルタスクを学習プロジェクトに登録します。
登録後、1 件ずつ description を改善していきましょう。

具体的な数値や状況は自由に想定して構いません。
この演習の評価ポイントは「構造が正しいか」です。」
```

既に同名タスクが存在する場合はスキップする。

---

## Step 3: 1 件ずつ改善

各タスクについて以下の手順で進める:

1. **問いかけ**: 「このタスク記述の問題点は何ですか？」
2. **受講者の回答を待つ**
3. ヒントが必要なら 4 観点のチェックリストを再提示:
   - 背景: 「なぜやるか」が書かれているか？
   - 対応内容: 具体的な変更箇所・操作が書かれているか？
   - 完了条件: Yes/No で判定できる検証項目があるか？
   - 制約: やらないこと・スコープ外が明記されているか？
4. **受講者に改善版を考えてもらう**
5. 改善版を確認し、フィードバックを提供
6. `upsert_task(taskId, description)` で改善版を学習プロジェクトに書き戻す

### 模範解答の提示ルール

- **1 件目（bad-1）**: 受講者の改善版を書き戻した後、**必ず** `bad-tasks.json` の `exampleFix.description` を模範解答として提示し、比較させる。受講者が困っていなくても提示する
- **2〜4 件目**: 受講者が困っている場合のみ模範解答を提示する

模範解答は「唯一の正解」ではないことを伝え、受講者自身の改善版と比較させる。

### ペース配分

- **最初の 3 件（bad-1, bad-3, bad-2）は丁寧に**: 問題点の特定 → 改善版作成 → フィードバック → 書き戻しをフルサイクルで実施
- **4 件目（bad-4）はテンポよく**: 問題点の確認と改善版を短めに進める
- **3 件完了時点で十分な学びが得られていると判断できる場合**（受講者が 4 観点を自力で適用できている）、4 件目はスキップしても構わない

---

## Step 4: /review-task で自己評価

改善版を `upsert_task` で書き戻した後、`/review-task` を実際に実行して品質を客観評価する。

```
「改善したタスク記述の品質を `/review-task` でチェックしてみましょう。
実際にレビュースキルを使って、改善前後の差を確認します。」
```

### 手順

1. 改善版が `upsert_task` で書き戻し済みであることを確認する
2. `/review-task` を実行し、レビュー結果を受講者に見せる
3. レビュー結果を読み合わせ、改善前後でどこが良くなったかを具体的にフィードバックする

`/review-task` は今後の開発でも使える実用ツールであることを伝える:

```
「/review-task はいつでも使えます。
自分でタスクを書いた後、提出前にセルフチェックする習慣をつけると効果的です。」
```

---

## Step 5: 振り返り

```
「4 件の改善を通じて、良いタスク記述のパターンが見えてきましたか？

共通するポイント:
1. 「おかしい」「遅い」「足りない」→ 具体的に何が、どう問題なのかを書く
2. 「直す」「改善する」「追加する」→ 具体的にどのファイル、どの関数、何をするかを書く
3. 完了条件は「〜できること」ではなく「〜した結果、〜が確認できること」
4. やらないことを明記することで、スコープの膨張を防ぐ

次の /learn-pc-write では、ゼロから自分でタスクを書きます。
改善するより難しいですが、ここで学んだパターンを活かしてください。」
```

---

## 注意事項

- 受講者自身に問題点を見つけてもらうことが重要。すぐに答えを教えない
- 模範解答は「唯一の正解」ではないことを伝える
- 模範解答を提示する際、技術用語の詳細（Lighthouse、Cognito、lazy loading 等）ではなく **構造のパターン** に注目するよう促す。「技術用語の詳細はここでは重要ではありません。4 観点がどう適用されているかに注目してください」
- 受講者の業務経験に応じて、より実践的な例を追加してもよい
- 演習タスクは必ず `/learn-setup` で作成した学習プロジェクトに登録する。本番プロジェクトには登録しないこと
