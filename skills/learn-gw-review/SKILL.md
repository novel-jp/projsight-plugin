# /learn-gw-review — レビュー→修正→マージ（06-03）

レビュー指摘への対応、修正コミット、マージ、complete_work までの一連のサイクルを体験する演習です。

**所要時間**: 約 45 分
**前提**: `/learn-gw-pr`（06-02）完了済み
**スキル対応**: Git Workflow（チーム開発フロー）

---

## Step 1: 導入

```
「コードレビューは品質を高めるだけでなく、チームの知識共有の場です。

AI 時代でもレビューの重要性は変わりません。
むしろ AI が高速にコードを生成するからこそ、
人間のレビューが品質のゲートキーパーとして機能します。

この演習では、レビュー指摘を受けてから完了までの全サイクルを体験します。」
```

---

## Step 2: レビュー指摘のシミュレーション

06-02 で作成した PR（またはこの演習用の新しい PR）を使う。

受講者に以下のレビュー指摘シナリオを提示する:

```
「あなたの PR に以下のレビューコメントが付きました:

1. 【改善要望】エラーハンドリングが不足している — 入力が不正な場合の処理を追加してください
2. 【質問】この実装方法を選んだ理由は？ — 代替手段との比較を PR description に追記してください
3. 【nit】変数名 `d` が分かりにくい — より説明的な名前にしてください」
```

### レビュー指摘の種類

| マーク                  | 意味       | 対応                                           |
| ----------------------- | ---------- | ---------------------------------------------- |
| **MUST** / 【改善要望】 | 修正必須   | 修正してコミット                               |
| **質問**                | 意図の確認 | 回答する（PR コメントまたは description 更新） |
| **nit**                 | 軽微な指摘 | 可能な限り対応（次回でも可）                   |

---

## Step 3: 修正の実施

レビュー指摘に対応する:

1. **コードの修正**: 指摘に基づいてコードを修正する
2. **コミット**: 修正をコミットする（修正内容が分かるコミットメッセージ）

```bash
git add .
git commit -m "fix: レビュー指摘対応 — エラーハンドリング追加、変数名改善"
```

3. **プッシュ**: 修正をプッシュする

```bash
git push
```

```
「レビュー対応のコツ:
- 各指摘に個別に対応し、何を変えたか説明する
- 『なぜそうしたか』の質問には丁寧に回答 — Intent Engineering の実践
- nit でも対応するとレビュアーの信頼を得られる」
```

---

## Step 4: マージと完了

レビュー対応後、マージから完了までの流れ:

1. **マージ**: PR をマージする

```bash
gh pr merge --squash
```

2. **ブランチの整理**: マージ済みブランチを削除

```bash
git checkout main && git pull && git branch -d <ブランチ名>
```

3. **ProjSight 完了**: `complete_work(taskId)` でタスクを完了にする

```
「complete_work の前に:
- PR がマージ済みであることを確認する
- 未登録の PR があれば attach_pr で紐づける
- complete_work は deliverable の進捗を自動計算する」
```

PR やリモートリポジトリがない学習環境の場合は、マージの手順を説明のみにし、`complete_work` の実行に集中する。

---

## Step 5: 振り返り

```
「レビュー→修正→マージ→完了のサイクルまとめ:

1. レビュー指摘を受ける — MUST / 質問 / nit を区別
2. 修正してコミット・プッシュ — 各指摘への対応を明確に
3. マージ — squash merge で履歴をきれいに
4. complete_work — ProjSight でタスク完了を記録

ポイント:
- レビューは『ダメ出し』ではなく『品質向上の協働作業』
- AI が書いたコードも人間が書いたコードも同じレビュープロセスを通す
- complete_work を忘れると、ProjSight の進捗が正確にならない

次の /learn-gw-cicd では、CI の結果の読み方と失敗時の対応を学びます。」
```

受講者のタスクを `complete_work(taskId)` で完了にする。
