# /learn-ce-maintain — DD 更新体験（02-04）

要件変更シナリオを受け、既存 DD の更新判断と実施を体験する演習です。

**所要時間**: 約 45 分
**前提**: `/learn-ce-dd`（02-03）完了済み
**スキル対応**: Context Engineering（情報環境の設計）

---

## Step 1: シナリオ提示

02-03 で作成した DD に対して 2 つの要件変更を提示する:

### シナリオ A（小規模変更）

```
「クライアントから『ページネーションの件数を 20 → 50 に変更してほしい』と依頼がありました。
この変更は DD にどう反映しますか？」
```

→ 判断: `upsert_dr(body)` で既存 DD を更新（設計の骨格は変わらない）

### シナリオ B（大規模変更）

```
「セキュリティレビューの結果、認証方式を JWT → セッション Cookie に変更することになりました。
この変更は DD にどう反映しますか？」
```

→ 判断: `add_design_doc(supersedes)` で新版を作成（構造変更）

---

## Step 2: 判断基準の解説

受講者に考えてもらった後、以下の判断基準を提示する:

| 変更の規模                   | 方法                                    | 判断基準                      |
| ---------------------------- | --------------------------------------- | ----------------------------- |
| **小規模**（数値変更、追記） | `upsert_dr(body)` で既存更新            | 設計の骨格は変わらない        |
| **大規模**（構造変更）       | `add_design_doc(supersedes)` で新版作成 | セクション追加/削除、方針転換 |
| **根本変更**                 | 新 DR + `add_design_doc(supersedes)`    | なぜ変えたかの記録も必要      |

```
「小規模なら上書き、大規模なら新版。
根本変更なら『なぜ変えたか』の DR も一緒に残す。
これが DD メンテナンスの基本判断です。」
```

---

## Step 3: 実施

受講者にシナリオ A と B の両方を実施してもらう。

### シナリオ A の実施

1. 02-03 で作成した DD を `get_dr` で取得
2. 該当箇所を修正
3. `upsert_dr(drId, body)` で更新

> **補足**: DD（設計ドキュメント）も DR エンティティの一種として管理されているため、更新には `upsert_dr` を使います。

### シナリオ B の実施

1. 認証方式変更に伴う DD 本文を書く
2. `add_design_doc(supersedes: 元のDD番号)` で新版を作成
3. 旧版が `superseded` ステータスになったことを確認

---

## Step 4: 振り返り

```
「DD は『生きたドキュメント』です。コードと DD が乖離すると、
AI が古い設計を参照して間違った実装をします。

DD の更新は Context Engineering の重要な一部です:
- コードを変えたら DD も変える
- 変更の規模に応じて『更新 vs 新版作成』を判断する
- supersedes で履歴を残すことで、設計の進化を追跡できる

次の /learn-ce-session では、セッション管理を学びます。」
```

受講者のタスクを `upsert_task(progressPct: 100)` で完了にする。
