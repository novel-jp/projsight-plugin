# /learn-ie-write — DR 作成（03-02）

技術選定テーマで DR を作成する演習です。

**所要時間**: 約 60 分
**前提**: `/learn-ie-analyze`（03-01）完了済み
**スキル対応**: Intent Engineering（目的と判断基準）

---

## Step 1: テーマ選択

以下から選択してもらう（または自分のプロジェクトから）:

| テーマ                          | 選択肢の例                |
| ------------------------------- | ------------------------- |
| **A**: 状態管理ライブラリの選定 | Redux vs Zustand vs Jotai |
| **B**: API 設計方針             | REST vs GraphQL           |
| **C**: テスト戦略               | ユニット重視 vs E2E 重視  |
| **D**: データベース選定         | RDB vs NoSQL              |
| **E**: 自由テーマ               | 受講者のプロジェクトから  |

```
「03-01 で良い DR と悪い DR の違いを学びました。
今度は自分で DR を書いてみましょう。

テーマは技術選定がおすすめです。
なじみのある技術でも、改めて『なぜそれを選ぶのか』を言語化してみてください。」
```

テーマを選んだら、選択肢の技術について **5 分程度リサーチする時間** を取る。知らない技術がある場合は AI に「各技術の特徴を教えて」と聞いても OK。

---

## Step 2: DR 作成ガイド

以下のフィールドを**必須**として誘導する:

| フィールド       | 説明                         | 注意点                         |
| ---------------- | ---------------------------- | ------------------------------ |
| **context**      | なぜこの判断が必要になったか | 「〜が必要」だけでなく背景まで |
| **decision**     | 何を選んだか、なぜか         | 結論 + 理由をセットで          |
| **alternatives** | 2 つ以上の代替案             | 各案に**却下理由**を記載       |
| **consequences** | ポジティブ・ネガティブ両方   | ネガティブを省略しない         |
| **制約条件**     | 技術的・ビジネス的制約       | 判断の前提となる条件           |

### DR テンプレート

`upsert_dr` の `body` に渡す Markdown は、以下の構成で書く。**各セクション 3〜5 行程度** が目安。03-01 で読んだ「良い DR」の構成を思い出しながら書いてみよう。

```markdown
## Context

<!-- なぜこの判断が必要になったか。プロジェクトの状況・課題を書く -->

新規プロジェクトで状態管理ライブラリを選定する必要がある。
チームメンバー3名、React 経験1年程度。中規模のダッシュボード SPA を開発予定。

## Decision

<!-- 何を選んだか + なぜその結論に至ったか -->

Zustand を採用する。学習コストの低さとバンドルサイズの小ささが、
チームの経験レベルとプロジェクト規模に最も適合するため。

## Alternatives

<!-- 2つ以上の代替案。各案に却下理由を必ず書く -->

### Redux Toolkit

業界標準だが、ボイラープレートが多く学習コストが高い。
3名の小規模チームには過剰。

### Jotai

アトミックな設計は魅力的だが、チーム内に経験者がおらず、
デバッグツールのエコシステムが Zustand より未成熟。

## Consequences

<!-- ポジティブ・ネガティブ両方を書く -->

**ポジティブ**: 導入が容易、バンドルサイズ小、TypeScript 相性良好
**ネガティブ**: 大規模化した場合にストア分割の設計指針が必要になる

## Constraints

<!-- 判断の前提となる技術的・ビジネス的制約 -->

- チームの React 経験が浅い（1年程度）
- 初期リリースまで2ヶ月の期限
- SSR は当面不要
```

```
「ヒント: 『なんとなく良さそうだから』は却下理由になりません。
具体的なメリット/デメリットの比較で判断してください。」
```

---

## Step 3: upsert_dr で登録

受講者が DR を書き終えたら、ProjSight に登録する。

AI に **「この内容で DR を登録して」** と伝えてください。AI が `upsert_dr` を実行します。projectId は AI が学習プロジェクトを自動特定するので、受講者が指定する必要はありません。

- type は `decision` を使う

---

## Step 4: /review-dr でセルフレビュー

`/review-dr` と入力して、作成した DR をレビューする。直前に登録した DR が自動的にレビュー対象になる。

- スコアは **10 点満点** で表示される
- スコアを確認し、**8 点未満**なら改善を提案する
- 特に以下の点を重点チェック:
  - 代替案が 2 つ以上あるか
  - 却下理由が具体的か
  - ネガティブな影響が記載されているか

改善が必要な場合は受講者に修正してもらい、再度 `/review-dr` でレビューする。

**改善ループの目安**: 2 回改善しても 8 点未満の場合は、レビュー指摘のうち**最も重要な 1 点**を修正して先に進んでください。完璧を目指すより、振り返りで学びを定着させることが大切です。

---

## Step 5: 振り返り

```
「DR を書くことで、『なんとなく選んだ』を『根拠を持って選んだ』に変えられます。

ポイント:
- DR は自分の判断力を証明するドキュメント
- 半年後に『なぜこうしたんだっけ？』と迷わなくなる
- AI にも DR を渡すことで、一貫性のある判断をしてもらえる
- 03-01 で見た悪い DR のパターンを、自分では書かないよう意識する」
```

振り返りの問いかけ:

```
「03-01 で分析した『悪い DR』のパターン（代替案なし、却下理由が曖昧、ネガティブ影響の省略など）を、
自分の DR では避けられましたか？ どの点が難しかったですか？」
```

受講者のタスクを `complete_work(taskId)` で完了にする。
