# /learn-ie-question — 質問管理（03-05）

不明点を構造化し、結論を記録するフローを体験する演習です。

**所要時間**: 約 30 分
**前提**: `/learn-ie-analyze`（03-01）完了済み
**スキル対応**: Intent Engineering（目的と判断基準）

---

## Step 1: 導入

```
「分からないことを『分からない』と構造化して伝えるのも Intent Engineering です。

曖昧な質問 → 曖昧な回答 → 判断できない → 時間を浪費
構造化された質問 → 的確な回答 → 判断できる → 前に進む

ProjSight の Question 機能を使って、不明点を管理するフローを体験します。」
```

---

## Step 2: 質問の作成

受講者のプロジェクトで「判断に迷うこと」を 1 つ特定する。この演習では **30 分で結論まで到達できるサイズ** の質問を選ぶよう促す。

**30 分で収まるサイズの目安**:

- 2 択の技術選定（例: SQLite vs JSON ファイル）
- Yes/No で答えられる方針判断（例: テストを書くべきか？）
- 優先順位の決定（例: 機能 A と機能 B のどちらを先に実装するか？）

特定が難しい場合のヒント:

- 「技術選定で迷っていることは？」
- 「チーム内で意見が分かれていることは？」
- 「ユーザーの要望で対応すべきか判断できないものは？」
- 「学習プロジェクトで、やり方を複数思いつくが決め手がないことは？」

> **テーマが浮かばない場合のサンプルシナリオ**
>
> 以下をそのまま使ってもかまいません（自分のテーマが出せた場合はそちらを優先）:
>
> _「学習プロジェクトのデータ永続化に SQLite と JSON ファイルのどちらを使うべきか？ SQLite は構造化クエリが使えるが学習コストがある。JSON ファイルは手軽だが検索性能に不安がある」_

### title と description の書き方

`upsert_question(title, description)` で登録する（`projectId` はスキルが自動設定します。この演習では `assigneeId` は省略します。チーム運用では質問の宛先として設定できます）。

**title**: 判断対象が一目でわかる短い疑問文で書く。
例: 「データ永続化に SQLite と JSON ファイルのどちらを使うか？」「リスト表示のパフォーマンス改善にページネーションと仮想スクロールのどちらを採用するか？」

body は以下の構造で書いてもらう:

```markdown
## 何が不明か

<!-- 具体的な疑問点 -->

## なぜ不明か

<!-- 情報不足？判断基準がない？意見が分かれている？ -->

## 自分の仮説

<!-- 現時点での推測があれば -->

## 判断に必要な情報

<!-- これが分かれば判断できる、という条件 -->
```

※ 返却された `questionId` を以降のステップで使用します。控えておいてください。

### 悪い質問と良い質問の比較

| 悪い質問                         | 良い質問                                                                                                                               |
| -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 「テスト書いた方がいいですか？」 | 「認証モジュールのユニットテストは必要か？現在カバレッジ 0%、リリース後のバグ報告が 3 件。E2E で代替可能だが、回帰テストの速度が課題」 |
| 「パフォーマンス大丈夫ですか？」 | 「リスト表示が 500 件超で 3 秒かかる。ページネーション vs 仮想スクロールのどちらで対応すべきか？ユーザーの 90% は 100 件以下」         |

## Step 3: コメントで議論

受講者・AI 双方のコメントを `add_question_comment(questionId, body)` で登録してください。これにより議論の全体が ProjSight に記録されます。受講者が発言したら受講者のコメントとして、AI が発言したら AI のコメントとして、それぞれ投稿します。

**時間配分**: Step 3 は **15 分を目安** に。収束チェックで結論が見えたら次へ進む。

受講者に以下のいずれかの形で議論してもらう:

- **メンター対話**（推奨）: 受講者の質問に対して AI がメンター役として回答・深掘り
- **自己対話**（上級者向け）: 受講者が自分で賛成/反対の立場を交互に書く。抽象思考に慣れている方向け

初回はメンター対話を推奨する。

### AI 向け進行指示（メンター対話モード）

メンター対話を選択した場合、AI は以下の手順で進行する。Step 2 で取得した `questionId` を保持し、Step 3〜4 で使用すること。

1. **初回コメント**: 受講者の仮説に対して反論または補足質問を `add_question_comment` で投稿する。「その仮説の根拠は？」「逆のケースではどうなる？」など、視野を広げる問いかけをする
2. **深掘り**: 受講者の返答（受講者のコメントも `add_question_comment` で投稿する）に対してさらに 1〜2 回深掘りする。トレードオフ（性能 vs 保守性、短期 vs 長期など）を意識させる
3. **論点整理**: 2〜3 往復したら、それまでの議論で明らかになった論点を整理して提示する
4. **結論促し**: 「ここまでの議論を踏まえて、結論を出せそうですか？」と受講者に判断を委ねる

### 自己対話モードの進め方

賛成と反対の立場を交互に `add_question_comment` で投稿する。以下は 1 往復分のサンプル:

> **賛成コメント例**（SQLite を推す立場）:
> 「SQLite を採用すべき。理由: 1) 複数条件での検索が SQL で簡潔に書ける、2) データ量が増えてもインデックスで対応可能、3) 将来 PostgreSQL 等への移行パスがある」
>
> **反対コメント例**（JSON ファイルを推す立場）:
> 「JSON ファイルで十分。理由: 1) 学習プロジェクトのデータ量は数百件程度、検索性能は問題にならない、2) セットアップ不要で即座に動く、3) SQLite のスキーマ管理は学習コストに見合わない」

2〜3 往復を目安に、論点が出尽くしたと感じたら収束チェックへ進む。

### 議論の収束チェック

2〜3 往復のコメントを積み重ねた後、AI は以下を確認する:

- 当初の仮説が **維持されたか、変更されたか**（どちらでも構わないが、理由が言語化されていること）
- **判断に必要な情報**（Step 2 で挙げたもの）が明らかになったか

いずれかが不十分であれば、「もう 1 往復だけ深掘りしましょう」と追加の議論を促す。

---

## Step 4: 結論と DR 化

1. `answer_question(questionId, answer)` で結論を記録する（Step 2 で取得した `questionId` を使用）
2. 以下のいずれかに当てはまれば `upsert_dr` で DR 化を推奨する:
   - 他のメンバーも同じ疑問を持ちそう
   - 3 ヶ月後に「なぜこう決めたか」を振り返る必要がある
   - 方針を変えると 1 日以上の手戻りが発生する

DR 化する場合は、03-02 で学んだ DR テンプレートを参照してください。Question の answer をベースに、代替案と却下理由を追記します。

```
「質問 → 議論 → 結論 → DR のフローは、チーム開発の意思決定プロセスそのものです。

Question: 『分からない』を記録する
Comment: 議論を蓄積する
Answer: 結論を確定する
DR: 重要な決定を永続化する」
```

## Step 5: 振り返り

```
「『分からない』を構造化する力は、メンターへの質問にも直接応用できます。

ポイント:
- 構造化された質問は、回答者の負荷を下げ、的確な回答を引き出す
- 質問 → 結論のフローを記録することで、同じ疑問を二度調べなくて済む
- 重要な結論は DR 化して、チームの知識資産にする

振り返りの問い:
- 今回作った質問テンプレート（何が不明か / なぜ不明か / 仮説 / 必要情報）は、Slack やチャットでの質問にもそのまま使えるか？

Intent Engineering セクション完了です！
03-01〜03-05 を通じて、『なぜ』を記録する力（DR 分析・DR 作成）と『分からない』を構造化する力（質問管理）が身につきました。

ここまでで、以下の力が身につきました:
- Context Engineering: AI の情報環境を設計する力
- Intent Engineering: 目的と判断基準を伝える力

次のセクション（Spec Engineering）では、AI が自律的に実行できる仕様を書く力を学びます。」
```

受講者のタスクを `complete_work(taskId)` で完了にする。
